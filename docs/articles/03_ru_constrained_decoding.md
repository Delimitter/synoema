# Type-guided constrained decoding: как заставить LLM не галлюцинировать код

## От GBNF-грамматик до type-directed generation: гарантии корректности вместо надежды

---

> **Для кого эта статья.** Если вы сталкивались с тем, что ChatGPT генерирует код, который не компилируется — эта статья объяснит, как это можно исключить полностью. Все технические термины объяснены в сносках и глоссарии в конце.

---

В предыдущих статьях мы показали, что сокращение токенов экономит деньги, энергию и compute. Но есть проблема серьёзнее: LLM генерирует **некорректный** код. И каждая попытка повторной генерации — это удвоение расхода токенов.

В этой статье мы рассмотрим, как **гарантировать** корректность сгенерированного кода — не статистически, а формально.

## Масштаб проблемы

Типовые ошибки составляют 33.6% всех отказов в LLM-сгенерированном коде (Mündler et al., PLDI 2025[^pldi]). Это не опечатки — это структурные ошибки: неправильные типы аргументов, несовместимые возвращаемые значения, обращение к несуществующим полям.

[^pldi]: **PLDI (Programming Language Design and Implementation)** — одна из самых престижных научных конференций по языкам программирования. Статьи проходят строгое рецензирование. Если результат опубликован в PLDI — ему можно доверять.

Когда LLM генерирует функцию сортировки, которая не компилируется, происходит одно из двух: человек правит руками (теряя время) или агент делает повторный запрос (тратя токены). В обоих случаях стоимость удваивается.

Но что если сама модель **физически не может** сгенерировать синтаксически невалидный код?

## Три уровня constraints

Гарантии корректности кода можно организовать в три уровня — от самого простого к самому мощному.

### Уровень 1: Грамматика (синтаксическая корректность)

На каждом шаге генерации определяется множество токенов, которые допустимы с точки зрения грамматики[^grammar] языка. Все остальные токены маскируются — их вероятность устанавливается в ноль.

[^grammar]: **Грамматика (формальная грамматика)** — набор правил, описывающих, какие последовательности символов являются допустимыми в языке. Например, правило «после `[` может идти число, идентификатор или `]`» — это часть грамматики. Грамматика Python сложная (сотни правил), грамматика JSON — простая (~10 правил).

Пример: если модель только что сгенерировала `[`, то следующим токеном может быть число, идентификатор, `]` или `[` — но не `+`, не `=`, не `)`.

**Инструменты:**

- **XGrammar[^xgrammar]** — дефолтный backend в SGLang[^sglang], vLLM, TensorRT-LLM[^trt]. Работает с контекстно-свободными грамматиками (CFG[^cfg]). Приближается к нулевому overhead.
- **Outlines[^outlines]** — structured generation через конечный автомат (FSM[^fsm]). Поддерживает regex и CFG.
- **llama.cpp[^llamacpp]** — встроенная поддержка GBNF-грамматик[^gbnf].
- **Guidance** (Microsoft) — шаблонная генерация с constraints.

[^xgrammar]: **XGrammar** — движок constrained decoding от команды MLC-AI. Стал стандартом де-факто для LLM inference. Его ключевая инновация — разделение словаря на «лёгкие» (80%+, проверяются заранее) и «тяжёлые» (20%, проверяются в runtime) токены, что даёт near-zero overhead.

[^sglang]: **SGLang** — open-source движок для LLM inference от UC Berkeley. Один из самых быстрых способов запуска LLM. Поддерживает constrained decoding через XGrammar «из коробки».

[^trt]: **TensorRT-LLM** — inference-движок от NVIDIA, оптимизированный для их GPU. Самый быстрый для NVIDIA-оборудования, но сложен в настройке.

[^cfg]: **CFG (Context-Free Grammar, контекстно-свободная грамматика)** — класс грамматик, где каждое правило имеет вид «символ → последовательность символов». Большинство языков программирования описываются CFG. JSON, XML, HTML, Python, JavaScript — все имеют CFG.

[^fsm]: **FSM (Finite State Machine, конечный автомат)** — математическая модель, которая в каждый момент находится в одном из конечных состояний и переходит между ними по входным символам. Используется для быстрой проверки, допустим ли следующий токен.

[^outlines]: **Outlines** — open-source библиотека для structured generation. Компилирует грамматику или regex в конечный автомат, который фильтрует токены на лету.

[^llamacpp]: **llama.cpp** — самый популярный open-source проект для запуска LLM на обычном железе (CPU, Mac M1/M2, слабые GPU). Написан на C++, работает без Python. Поддерживает GBNF-грамматики для constrained decoding.

[^gbnf]: **GBNF (GGML BNF)** — формат описания грамматик, используемый в llama.cpp. Расширение стандартной формы Бэкуса–Наура (BNF). Похож на EBNF, но с синтаксисом, адаптированным для llama.cpp. Пример: `expr ::= number | expr "+" expr`.

Результат: **100% синтаксическая корректность.** Каждый сгенерированный фрагмент — валидная программа.

### Уровень 2: Типы (семантическая корректность)

Грамматика гарантирует, что `f x = x + 1` синтаксически валидна. Но не гарантирует, что `x` — число. Type-constrained decoding[^tcd] добавляет второй слой: на каждом шаге допускаются только токены, совместимые с текущим типовым контекстом.

[^tcd]: **Type-constrained decoding (генерация с учётом типов)** — расширение constrained decoding, где помимо грамматики проверяются типы. Если функция ожидает `Int`, модель не может подставить `String`. Требует type inference — автоматического вывода типов компилятором.

Mündler et al. (PLDI 2025) показали, что type-constrained decoding сокращает ошибки компиляции на **74.8%** по сравнению с 9.0% для syntax-only constraints.

Для этого нужна система типов с выводом[^inference] — чтобы в каждой точке генерации компилятор мог определить допустимые типы без явных аннотаций.

[^inference]: **Type inference (вывод типов)** — способность компилятора определять типы выражений автоматически, без явных аннотаций от программиста. Вместо `int add(int x, int y)` (как в C) пишем просто `add x y = x + y`, и компилятор сам определяет, что это `Int → Int → Int`. Самый мощный алгоритм вывода типов — Hindley-Milner, используемый в Haskell и ML.

### Уровень 3: Спецификация (логическая корректность)

Самый мощный уровень: constraints на основе формальной спецификации. Функция сортировки не просто имеет правильный тип `List Int → List Int` — она действительно сортирует: выход — перестановка входа, и каждый элемент не больше следующего.

Это область активных исследований (dependent types, refinement types). В продакшн-инструментах пока не реализовано.

## Как работает XGrammar

Ключевая оптимизация XGrammar — **разделение словаря на два класса:**

**Context-independent токены (~80%+).** Их допустимость определяется на этапе препроцессинга, до начала генерации. Для каждого состояния грамматики заранее вычисляется битовая маска допустимых токенов. Это lookup-таблица — O(1) на токен.

**Context-dependent токены (~20%).** Их допустимость зависит от текущего состояния PDA[^pda]. Проверяются в runtime, но их мало.

[^pda]: **PDA (Pushdown Automaton, автомат с магазинной памятью)** — расширение конечного автомата со стеком. Нужен для разбора грамматик с вложенностью (скобки, блоки кода). Обычный FSM не может считать скобки — PDA может.

Результат: **near-zero overhead.** Включение constrained decoding не добавляет заметного overhead к TPOT[^tpot].

[^tpot]: **TPOT (Time Per Output Token)** — время генерации одного выходного токена. Основная метрика скорости LLM inference. Для GPT-4 это ~20–30 мс, для небольших моделей на мощном GPU — 5–10 мс.

## BPE misalignment ломает constrained decoding

И вот здесь дизайн языка становится критически важным.

Когда грамматика языка не выровнена по BPE-границам, constrained decoding сталкивается с проблемой **мостовых токенов** (bridge tokens).

Мостовой токен — это BPE-токен, который «перекрывает» границу между двумя грамматическими символами. Например, если BPE-токенизатор объединяет ` if` (пробел + ключевое слово) в один токен, то constrained decoding engine должен решить: допустим ли этот токен?

**Domino** (ICML 2024[^icml]) показал, что мостовые токены искажают вероятностное распределение модели. **Grammar-Aligned Decoding** (NeurIPS 2024[^neurips]) формализовал эту проблему теоретически и предложил решение — но с дополнительным overhead.

[^icml]: **ICML (International Conference on Machine Learning)** — одна из трёх топовых конференций по машинному обучению (вместе с NeurIPS и ICLR). Публикация в ICML — знак высокого качества исследования.

[^neurips]: **NeurIPS (Neural Information Processing Systems)** — крупнейшая конференция по AI и машинному обучению. ~15 000 участников ежегодно. Публикация в NeurIPS — золотой стандарт для ML-исследований.

Если язык спроектирован так, что мостовые токены **не возникают** — то есть каждый грамматический символ совпадает с одним BPE-токеном — проблема исчезает полностью.

## Детерминированная CFG = нулевой overhead

Ещё один фактор: **тип грамматики** языка.

**Недетерминированная CFG** — при разборе текущего токена невозможно однозначно определить, какое правило применять. Нужен backtracking[^bt] или параллельное исследование. Это дорого.

[^bt]: **Backtracking (откат)** — метод разбора, при котором парсер пробует одно правило, и если оно не подошло — откатывается и пробует другое. Медленный, потому что одну и ту же часть текста можно разбирать многократно.

**Детерминированная CFG (DCFG)[^dcfg]** — при разборе каждого токена применимо ровно одно правило. Грамматика компилируется в конечный автомат и constraints вычисляются в замкнутой форме. Нет backtracking. Нет ambiguity.

[^dcfg]: **DCFG (Deterministic Context-Free Grammar)** — подкласс CFG, где на каждом шаге разбора однозначно определяется следующее правило. Компилируется в эффективный автомат. Большинство реальных языков программирования — DCFG (или близки к ней). Python — формально не DCFG из-за отступов, но с offside rule приближается.

Tian et al. (CoLM 2024[^colm]) доказали, что для DCFG constrained decoding компилируется в **замкнутой форме** — overhead близок к нулю.

[^colm]: **CoLM (Collocated with Language Models)** — новая конференция, посвящённая пересечению языковых моделей и формальных методов. Фокус на том, как теория компиляторов может улучшить LLM.

Если язык спроектирован с DCFG-грамматикой и BPE-aligned операторами, то constrained decoding становится **бесплатным**: нулевой overhead + нулевые мостовые токены.

## Практика: GBNF-грамматика

Как это выглядит на практике? Вот фрагмент GBNF-грамматики для BPE-оптимизированного языка:

```gbnf
root    ::= program
program ::= (decl newline)* decl newline?
decl    ::= func-def | type-sig | type-def

func-def ::= lower-id ws (pattern ws)* "=" ws expr
expr     ::= cond-expr | pipe-expr
cond-expr ::= "?" ws expr ws "->" ws expr ws ":" ws expr
```

Подключение к SGLang — одна строка:

```python
response = client.chat.completions.create(
    model="default",
    messages=[{"role": "user", "content": "Write factorial"}],
    extra_body={"ebnf": open("synoema.gbnf").read()},
)
# Результат: ГАРАНТИРОВАННО синтаксически валидный код
```

Или к llama.cpp:

```bash
./main -m model.gguf --grammar-file synoema.gbnf \
  -p "-- Quicksort:" -n 128 --temp 0.2
```

## Экономический эффект

Суммарный эффект трёх рычагов:

| Рычаг | Механизм | Экономия |
|-------|----------|----------|
| BPE-aligned грамматика | 46% меньше токенов | -46% прямая |
| Квадратичный attention | 54% длины → 29% стоимости | -71% на attention |
| Constrained decoding | 0 невалидного кода → 0 повторов | -10–30% |
| Type constraints | -74.8% типовых ошибок | -5–15% дополнительно |

Суммарная экономия: **50–70%** стоимости и энергии по сравнению с неоптимизированной генерацией Python-кода.

## Что дальше

В следующей статье мы представим **Synoema** — язык программирования, в котором все три рычага реализованы: BPE-aligned грамматика с 33 одно-токенными операторами, Hindley-Milner[^hm] type inference для type-guided constraints, и Cranelift[^cranelift] JIT для нативной скорости исполнения.

[^hm]: **Hindley-Milner** — алгоритм автоматического вывода типов, разработанный в 1960–80-х годах. Позволяет компилятору определить типы всех выражений в программе **без единой аннотации** от программиста. Используется в Haskell, OCaml, F#, Elm. Подробнее — в пятой статье серии.

[^cranelift]: **Cranelift** — компилятор-backend, написанный на Rust. Превращает промежуточное представление (IR) в нативный машинный код (x86-64, ARM). Альтернатива LLVM: компилирует в 10 раз быстрее, хотя даёт код примерно на 14% медленнее. Идеален для JIT-компиляции, где скорость компиляции важнее максимальной оптимизации.
