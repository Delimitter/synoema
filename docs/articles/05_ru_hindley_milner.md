# Hindley-Milner для LLM: вывод типов без аннотаций

## Полиморфная типизация: меньше токенов, больше гарантий

---

> **Для кого эта статья.** Если вы задавались вопросом, можно ли иметь строгую типизацию без многословных аннотаций как в Java или TypeScript — ответ: да. Эта статья объяснит, как, и почему это критически важно для LLM.

---

В предыдущих статьях мы рассмотрели три рычага оптимизации: сжатие представления (46% экономии), constrained decoding (100% синтаксическая корректность) и JIT-компиляция (4.4× быстрее Python). Осталось одно слабое звено: **типовые ошибки**.

33.6% всех отказов LLM-сгенерированного кода — это ошибки типов. Можно ли их устранить, не заставляя LLM генерировать аннотации типов?

## Проблема: типы стоят токенов

Типовые аннотации — это дополнительные токены, которые LLM должен генерировать.

```typescript
// TypeScript: 50% токенов — аннотации типов
function add(a: number, b: number): number {
    return a + b;
}
// ~15 токенов, из них ~7 — типовые аннотации
```

```java
// Java: ещё хуже
public static int add(int a, int b) {
    return a + b;
}
// ~16 токенов, из них ~8 — типовые
```

Каждая аннотация — это:
- Дополнительные токены для LLM (стоимость генерации).
- Дополнительное место в контексте (квадратичная стоимость attention).
- Дополнительная возможность для ошибки (LLM может написать `string` вместо `number`).

Идеальный вариант: **100% типовых гарантий при нуле типовых аннотаций.**

## Hindley-Milner: решение 1960-х для проблемы 2020-х

Алгоритм Hindley-Milner[^hm] — это система вывода типов, разработанная математиками Хиндли (1969) и Милнером (1978). Она позволяет компилятору **автоматически** определить тип каждого выражения в программе, **не требуя ни одной аннотации**.

[^hm]: **Hindley-Milner (HM)** — алгоритм, который по исходному коду без единой типовой аннотации определяет наиболее общий тип каждого выражения. Математически доказано, что HM всегда находит **principal type** (самый общий тип) — или сообщает об ошибке. Используется в Haskell, OCaml, F#, Elm, Standard ML.

```
-- Synoema: ноль аннотаций, 100% type safety
add x y = x + y
-- Компилятор выводит: Int → Int → Int

id x = x
-- Компилятор выводит: ∀a. a → a (полиморфный[^poly] тип)

map f [] = []
map f (x:xs) = f x : map f xs
-- Компилятор выводит: ∀a b. (a → b) → List a → List b
```

[^poly]: **Полиморфный тип (polymorphic type)** — тип, который работает с разными типами данных. Функция `id x = x` принимает значение любого типа и возвращает его же. Записывается `∀a. a → a`: «для любого типа `a`, принимаю `a` и возвращаю `a`». Это как generics в Java/TypeScript, но без угловых скобок и без явного указания.

LLM не нужно генерировать ни одного типового токена. Компилятор знает типы всех выражений.

## Как это работает

Алгоритм HM (также называемый Algorithm W[^algw]) работает в три шага:

[^algw]: **Algorithm W** — конкретная реализация вывода типов Hindley-Milner, предложенная Дамасом и Милнером в 1982 году. «W» — от «well-typing». Обходит AST дерева программы снизу вверх, собирая ограничения на типы и решая их через унификацию.

**Шаг 1: Генерация ограничений.** Для каждого выражения компилятор создаёт типовую переменную[^typevar] и записывает ограничения. Например, для `x + y`:
- `x : τ₁`, `y : τ₂`, `(+) : Int → Int → Int`
- Ограничение: `τ₁ = Int`, `τ₂ = Int`

[^typevar]: **Типовая переменная (type variable)** — «заглушка» для ещё неизвестного типа. Обозначается греческими буквами (τ, α, β) или латинскими (a, b). В ходе вывода типов заглушки заменяются конкретными типами через процесс унификации.

**Шаг 2: Унификация[^unification].** Система ограничений решается: типовые переменные заменяются конкретными типами. Если ограничения противоречивы — компилятор сообщает об ошибке.

[^unification]: **Унификация** — процесс нахождения подстановки, которая делает два типовых выражения равными. Например, унификация `List a` и `List Int` даёт `a = Int`. Унификация `Int` и `String` — ошибка. Это основа вывода типов: компилятор собирает ограничения и «решает» их через унификацию.

**Шаг 3: Обобщение[^generalize].** Оставшиеся свободные переменные обобщаются: `id : a → a` становится `id : ∀a. a → a` — функция работает с любым типом.

[^generalize]: **Обобщение (generalization)** — процесс превращения конкретного типа в полиморфный. Если после вывода типов переменная `a` осталась свободной (не была привязана к конкретному типу), она обобщается: `a → a` становится `∀a. a → a`. Это позволяет использовать `id` как `id 42` (Int) и `id true` (Bool) в одной программе.

Ключевое свойство: **HM всегда находит наиболее общий тип.** Не нужно угадывать или указывать — алгоритм гарантированно находит правильный ответ (или сообщает об ошибке).

## Let-polymorphism: почему это мощнее generics

В Synoema реализован let-polymorphism[^letpoly] — расширение HM, которое позволяет одной функции иметь разные типы в разных местах использования:

[^letpoly]: **Let-polymorphism** — механизм, при котором переменная, определённая через `let` (или `=` в Synoema), получает полиморфный тип. Это значит: `id` может быть одновременно `Int → Int` и `Bool → Bool` в одной программе. Без let-polymorphism `id` был бы зафиксирован в одном типе.

```
id x = x

main =
  a = id 42        -- id используется как Int → Int
  b = id true       -- id используется как Bool → Bool
  a                 -- Нет ошибки! id полиморфна.
```

Для сравнения: в Go (до версии 1.18) это было невозможно без `interface{}`. В Python это «работает» через duck typing — но без гарантий: ошибка обнаружится только в runtime.

## Взаимодействие с constrained decoding

Вот где HM становится по-настоящему мощным для LLM.

На каждом шаге генерации кода компилятор может определить **допустимые типы** следующего выражения. Это создаёт второй слой constraints поверх грамматики:

```
-- LLM генерирует: map ??? [1 2 3]
-- Компилятор знает: ??? : Int → τ  (функция от Int)
-- Допустимы: \x -> x + 1, \x -> x * 2, show
-- Недопустимы: \x -> x ++ "hello" (String ≠ Int)
```

Грамматический constraint говорит: «здесь может быть выражение».
Типовой constraint добавляет: «и это выражение должно иметь тип `Int → τ`».

Вместе они сужают пространство допустимых продолжений на порядки — LLM нужно выбирать из значительно меньшего множества, что снижает вероятность ошибки и ускоряет генерацию.

| Подход | Типовые гарантии | Токены на типы | Ошибки в runtime |
|--------|-----------------|----------------|-----------------|

| Java | Есть | ~40–60% кода | Мало |

Synoema — единственный вариант, где **гарантии максимальны, а стоимость нулевая**.

## Глоссарий

| Термин | Объяснение |
|--------|-----------|
| **Hindley-Milner (HM)** | Алгоритм автоматического вывода типов. Определяет тип каждого выражения без аннотаций |
| **Algorithm W** | Конкретная реализация HM. Обходит AST снизу вверх, собирая ограничения |
| **Типовая переменная** | «Заглушка» для неизвестного типа, которая заменяется через унификацию |
| **Унификация** | Процесс приравнивания двух типовых выражений. Основа вывода типов |
| **Обобщение** | Превращение конкретного типа в полиморфный: `a → a` → `∀a. a → a` |
| **Let-polymorphism** | Механизм, позволяющий одной функции иметь разные типы в разных контекстах |
| **Полиморфный тип** | Тип, работающий с разными типами данных. `∀a. a → a` — «для любого типа» |
| **Principal type** | Наиболее общий тип выражения. HM всегда находит principal type |
| **Duck typing** | Принцип Python: типы не проверяются до момента исполнения |
| **Generics** | Обобщённые типы в Java/TypeScript. HM — более мощная версия той же идеи |
