# Анатомия BPE: почему Python тратит 46% токенов впустую

## Как работает BPE-токенизация и что она означает для дизайна языков программирования

В прошлой статье мы установили, что стоимость inference растёт квадратично с количеством токенов. Логичный вопрос: можно ли сократить количество токенов, не теряя семантику?

Чтобы ответить на него, нужно понять, как LLM видит код. Не как текст — как последовательность токенов. И между тем, как программист видит `def factorial(n):`, и тем, как это видит GPT-4, — пропасть.

## Как работает BPE

BPE (Byte Pair Encoding)[^bpe] — алгоритм, который превращает текст в последовательность целых чисел (токенов). Он лежит в основе всех современных LLM: GPT-4 использует словарь cl100k_base[^cl100k], Claude — модифицированный BPE, Llama — SentencePiece[^sentencepiece] BPE.

[^bpe]: **BPE (Byte Pair Encoding)** — «кодирование парами байтов». Алгоритм сжатия текста, придуманный в 1994 году и адаптированный для LLM. Суть: находим самые частые пары символов в огромном корпусе текста и объединяем их в новый символ. Повторяем ~100 000 раз. Результат — словарь «подслов», которыми модель думает.

[^cl100k]: **cl100k_base** — конкретный словарь BPE-токенов, используемый в GPT-4 и Claude. Содержит ~100 000 токенов. Обучен в основном на английском тексте из интернета. Название расшифровывается как «cl» (ChatGPT Language?) + «100k» (100 тысяч токенов) + «base» (базовый). GPT-4o использует более новый словарь o200k_base с ~200 000 токенами.

[^sentencepiece]: **SentencePiece** — альтернативная реализация BPE от Google, используемая в Llama и других открытых моделях. Работает на уровне Unicode-символов вместо байтов, что лучше для неанглийских языков.

Алгоритм прост:

1. Начинаем с алфавита из отдельных байтов (256 символов).
2. Находим самую частую пару соседних символов в корпусе[^corpus].
3. Создаём новый символ для этой пары и добавляем в словарь.
4. Повторяем шаги 2–3 до нужного размера словаря (~100K).

[^corpus]: **Корпус (corpus)** — огромный набор текстов, на котором обучается BPE-словарь (и сама LLM). Включает веб-страницы, книги, статьи, код с GitHub. Обычно сотни миллиардов токенов. Код составляет лишь ~5–15% типичного корпуса — поэтому BPE оптимизирован под английскую прозу, а не под синтаксис Python.

Результат — словарь из ~100 000 «подслов» переменной длины. Короткие и частые слова (`the`, `is`, `def`) кодируются одним токеном. Редкие слова разбиваются на части: `tokenization` → `token` + `ization`.

**Ключевое свойство:** BPE обучается на **естественном языке**, а не на коде. Поэтому он оптимизирован под английскую прозу, а не под синтаксис Python.

## Проблема несовпадения

Грамматические единицы языка программирования — операторы, ключевые слова, разделители — **не совпадают** с границами BPE-токенов. Это создаёт два типа потерь.

**Тип 1: Избыточные токены на синтаксис.**

Возьмём простую функцию на Python:

```python
def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)
```

BPE (cl100k_base) разбивает это на **29 токенов**. Из них семантически значимы только `factorial`, `n`, `0`, `1`, `*`, `-`. Остальные 23 токена — синтаксический оверхед: `def`, пробелы, `(`, `)`, `:`, `if`, `==`, `return` (дважды), отступы, переносы строк.

Эквивалентная программа на функциональном языке с минимальным синтаксисом:

```
fac 0 = 1
fac n = n * fac (n - 1)
```

**16 токенов.** Та же семантика. На 45% меньше.

**Тип 2: Мостовые токены (bridge tokens)[^bridge].**

[^bridge]: **Мостовой токен (bridge token)** — BPE-токен, который «перекрывает» границу между двумя грамматическими символами. Например, BPE может объединить пробел и ключевое слово ` if` в один токен. Это создаёт проблемы для constrained decoding — движку нужно «разрезать» токен на части, что искажает вероятности модели. Подробнее — в третьей статье серии.

Иногда один BPE-токен перекрывает границу между двумя грамматическими символами. Например, строка ` "name"` в JSON может стать одним токеном, хотя грамматически это пробел + кавычка + идентификатор + кавычка. Это создаёт проблемы для constrained decoding[^cd] — движку нужно «разрезать» токен на части, что искажает распределение вероятностей модели.

[^cd]: **Constrained decoding (ограниченная генерация)** — технология, которая на каждом шаге генерации запрещает модели выбирать недопустимые токены. Гарантирует, что сгенерированный код синтаксически валиден. Подробно разбирается в третьей статье серии.

## Бенчмарк: 12 программ, 3 языка

Чтобы измерить масштаб потерь, я сравнил количество токенов для эквивалентных программ на трёх языках: Python, Haskell[^haskell] и оптимизированный язык, где каждый оператор — ровно один BPE-токен.

[^haskell]: **Haskell** — функциональный язык программирования с минимальным синтаксисом. Используется как эталон «краткости» среди существующих языков. `fac 0 = 1` на Haskell и на оптимизированном языке выглядят почти одинаково, но оптимизированный язык дополнительно учитывает BPE-границы.

Программы выбраны типичные: factorial, map, quicksort, fizzbuzz, fibonacci, filter, sum, length, reverse, compose, maximum, zip.

| Программа | Оптимизированный | Python | Haskell | Экономия vs Python |
|-----------|-----------------|--------|---------|-------------------|
| Factorial | 16 | 29 | 16 | 45% |
| Map | 20 | 42 | 23 | 52% |
| QuickSort | 51 | 83 | 54 | 39% |
| FizzBuzz | 44 | 64 | 49 | 31% |
| Filter | 27 | 67 | 36 | 60% |
| Fibonacci | 26 | 46 | 26 | 43% |
| Sum | 16 | 33 | 19 | 52% |
| Length | 16 | 30 | 19 | 47% |
| Reverse | 18 | 35 | 18 | 49% |
| Compose | 38 | 75 | 39 | 49% |
| Maximum | 28 | 58 | 37 | 52% |
| Zip | 32 | 53 | 37 | 40% |
| **Итого** | **332** | **615** | **373** | **46%** |

**Оптимизированный язык тратит на 46% меньше токенов, чем Python**, и на 11% меньше, чем Haskell.

С учётом квадратичной стоимости attention: 46% меньше токенов ≈ **71% меньше вычислений** на attention-слоях.

## Откуда берётся экономия

Разберём по конструкциям, где именно теряются токены.

### Определение функции

```python
# Python: 6 токенов оболочки
def add(x, y):
    return x + y

# Оптимизированный: 0 токенов оболочки
add x y = x + y
```

Python тратит `def`, `(`, `,`, `)`, `:`, `return` — 6 токенов чистого синтаксиса. Оптимизированный язык использует только `=`.

### Условие

```python
# Python: 6 токенов оверхеда
if x > 0:
    return x
else:
    return -x

# Оптимизированный: 3 токена
? x > 0 -> x : -x
```

### Списки

```python
# Python: N-1 запятых
[1, 2, 3, 4, 5]  # 9 токенов

# Оптимизированный: 0 запятых
[1 2 3 4 5]       # 7 токенов
```

В списке из 100 элементов — 99 лишних токенов.

### Pattern matching[^pm]

[^pm]: **Pattern matching (сопоставление с образцом)** — способ определения функции через «примеры». Вместо `if n == 0: return 1` пишем `fac 0 = 1` — буквально: «факториал нуля — это единица». Компилятор сам генерирует проверку. Это короче, яснее и исключает целый класс ошибок.

```python
# Python: нет нативного pattern matching для рекурсии
def fac(n):
    if n == 0:
        return 1
    return n * fac(n - 1)
# 29 токенов

# Оптимизированный
fac 0 = 1
fac n = n * fac (n - 1)
# 16 токенов
```

## BPE-aligned grammar

Описанная выше экономия — не просто результат «краткого синтаксиса». Она достигается через целенаправленный дизайн грамматики с учётом свойств BPE-токенизатора.

**Принцип BPE-aligned grammar[^aligned]:** каждый оператор языка должен быть ровно одним BPE-токеном.

[^aligned]: **BPE-aligned grammar (грамматика, выровненная по BPE)** — принцип проектирования языка, при котором каждый оператор, ключевое слово и разделитель кодируется ровно в один BPE-токен. Это означает: нет «лишних» токенов на синтаксис и нет мостовых токенов. Обычные языки не учитывают BPE при дизайне — они создавались задолго до LLM.

Это не тривиальное условие. Например, оператор `<=>` — это 2 BPE-токена (`<=` + `>`), а `!=` — 1 BPE-токен. Оператор `&&` — 1 токен, но `and` — тоже 1 токен (и занимает 3 символа вместо 2).

Для оптимизированного языка проверены все 33 оператора — каждый из них кодируется ровно в 1 BPE-токен на cl100k_base (GPT-4/Claude) и o200k_base[^o200k] (GPT-4o):

[^o200k]: **o200k_base** — более новый BPE-словарь, используемый в GPT-4o. Содержит ~200 000 токенов (вдвое больше, чем cl100k_base). Лучше покрывает код и неанглийские языки, но базовые принципы те же.

```
Два символа, 1 токен:  -> <- |> ++ >> == != <= >= && || ..
Один символ, 1 токен:  ? : . = @ | \ _ , + - * / % < >
Разделители, 1 токен:  ( ) [ ]
```

Это не совпадение — это **constraint на дизайн грамматики**. Если оператор не укладывается в один BPE-токен, он заменяется на тот, который укладывается.

## Что это означает для LLM

Когда LLM генерирует код на оптимизированном языке вместо Python:

1. **Генерирует на 46% меньше токенов** → быстрее, дешевле, меньше энергии.
2. **Тратит на 71% меньше attention** → может работать с бо́льшими кодовыми базами в том же контексте.
3. **Не тратит токены на «мосты»** → BPE-aligned грамматика не создаёт мостовых токенов, что упрощает constrained decoding.
4. **Не может «болтать»** → минимальный синтаксис не даёт раздуть код без добавления семантики.

## Что дальше

Краткость — необходимое, но не достаточное условие. Можно сократить токены обфускацией, но это убьёт корректность. В следующей статье мы рассмотрим **constrained decoding** — технологию, которая гарантирует 100% синтаксическую корректность сгенерированного кода. И покажем, почему BPE-aligned грамматика делает constrained decoding **бесплатным**.
