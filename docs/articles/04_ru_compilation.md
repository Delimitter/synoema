# Компиляция для LLM: зачем языку моделей нужен нативный код

## Cranelift JIT, 4.4x быстрее Python, и почему это критично для AI-агентов

---

> **Для кого эта статья.** Если вы строите AI-агентов, которые генерируют и исполняют код, или хотите понять, почему скомпилированный код из LLM — не фантастика, а работающая технология — читайте. Все термины объяснены в сносках и глоссарии.

---

В предыдущих статьях мы показали, как сократить количество токенов на 46% и гарантировать синтаксическую корректность. Но есть третья проблема: сгенерированный код должен не только быть коротким и корректным — он должен быть **быстрым**.

## Контекст: LLM-агенты пишут и запускают код

Claude Code, Cursor, Devin, OpenAI Codex — все эти инструменты не просто генерируют код. Они его **исполняют**: запускают тесты, обрабатывают данные, вызывают API. Цикл «сгенерировать → запустить → проанализировать результат → повторить» — это основа agentic workflows[^agentic].

[^agentic]: **Agentic workflows (агентные рабочие процессы)** — подход, при котором LLM действует как автономный «агент»: получает задачу, разбивает на шаги, пишет код, запускает его, анализирует результат и корректирует. В отличие от простого чат-бота, агент может выполнять многошаговые задачи самостоятельно.

Проблема: почти все агенты генерируют **Python**. А Python — интерпретируемый язык[^interpreted].

[^interpreted]: **Интерпретируемый язык** — язык, код которого исполняется «построчно» интерпретатором, без предварительной компиляции в машинный код. Python, JavaScript, Ruby — интерпретируемые. C, Rust, Go — компилируемые (код сначала превращается в машинные инструкции, потом исполняется процессором напрямую). Интерпретируемые языки проще, но в 10–100 раз медленнее компилируемых.

Это означает:
- Каждый запуск проходит через интерпретатор CPython — медленный, однопоточный.
- Нет оптимизации кода: Python не знает типы до момента исполнения (duck typing[^duck]).
- Для серьёзных вычислений нужны внешние библиотеки на C (NumPy, pandas), что добавляет зависимости и сложность.

[^duck]: **Duck typing** — принцип Python: «если оно ходит как утка и крякает как утка — это утка». Тип переменной не объявляется и не проверяется заранее — ошибка типа обнаруживается только при исполнении. Это удобно для прототипов, но опасно для продакшн-кода и делает невозможной компиляцию в эффективный машинный код.

Для одноразовых скриптов — нормально. Для серверов, data pipeline'ов, realtime-обработки — нет.

## Решение: JIT-компиляция

Что если сгенерированный LLM код **компилируется в нативный машинный код** за миллисекунды и исполняется на скорости C?

Именно это делает JIT-компиляция[^jit]:

[^jit]: **JIT (Just-In-Time) компиляция** — компиляция кода в машинные инструкции непосредственно перед исполнением, «на лету». В отличие от AOT-компиляции (Ahead-Of-Time, как в C/Rust), где код компилируется заранее, JIT компилирует в момент запуска. Преимущество: не нужен отдельный шаг сборки. LLM генерирует код → JIT компилирует за миллисекунды → нативная скорость исполнения.

```
LLM генерирует код (.sno)
    ↓
Парсер → AST[^ast] → Type Check → Core IR[^coreir]
    ↓
Cranelift JIT → нативный x86-64 машинный код
    ↓
Исполнение на скорости C/Rust (без интерпретатора)
```

[^ast]: **AST (Abstract Syntax Tree, абстрактное синтаксическое дерево)** — структура данных, представляющая программу в виде дерева. Парсер превращает текст кода в AST, после чего компилятор работает с деревом, а не с текстом. Например, `2 + 3 * 4` превращается в дерево, где `*` — дочерний узел `+`.

[^coreir]: **Core IR (Intermediate Representation, промежуточное представление)** — упрощённый «внутренний язык» компилятора. Исходный код сначала превращается в Core IR (убираются синтаксический сахар, паттерн-матчинг раскрывается в case-выражения), а затем Core IR компилируется в машинный код. Это System F — формализм из теории типов.

Весь цикл — от текста до нативного кода — занимает **< 100 мс**. Для сравнения: `pip install numpy` занимает секунды, а `import pandas` — сотни миллисекунд.

## Почему Cranelift, а не LLVM

Для JIT-компиляции нам нужен backend-компилятор[^backend] — инструмент, который превращает Core IR в машинный код. Два основных варианта:

[^backend]: **Backend-компилятор** — часть компилятора, которая генерирует финальный машинный код. Frontend (парсер, type checker) работает с исходным кодом, backend — с машинными инструкциями. LLVM и Cranelift — два самых популярных backend'а.

**LLVM[^llvm]** — промышленный стандарт. Используется в Clang (C/C++), Rust, Swift, Julia. Генерирует очень быстрый код, но сам компилирует **медленно**: десятки миллисекунд на функцию. Написан на C++, тянет гигабайты зависимостей.

[^llvm]: **LLVM (Low Level Virtual Machine)** — фреймворк для создания компиляторов, начатый в 2003 году Крисом Латтнером. Стал стандартом де-факто: на нём основаны компиляторы C/C++ (Clang), Rust, Swift, Julia, Zig и десятки других языков. Генерирует максимально оптимизированный машинный код, но сложен в использовании и медленно компилирует.

**Cranelift[^cranelift]** — написан на чистом Rust. Компилирует в **10 раз быстрее**, чем LLVM. Генерирует код ~86% от качества LLVM. Идеален для JIT: скорость компиляции важнее пиковой оптимизации.

[^cranelift]: **Cranelift** — backend-компилятор, созданный Bytecode Alliance (Mozilla, Fastly, Intel). Написан целиком на Rust. Используется в Wasmtime (WebAssembly runtime) и как JIT-backend для нескольких языков. Компилирует x86-64, ARM64, RISC-V.

| Критерий | LLVM | Cranelift |
|----------|------|-----------|
| Язык | C++ | Rust |
| Скорость компиляции | 1× | **10×** |
| Качество кода | 100% | ~86% |
| Зависимости | Гигабайты | `cargo build` |
| Идеален для | AOT-компиляции | **JIT-компиляции** |

Для agentic workflow важна **скорость цикла**: LLM генерирует → компилирует → исполняет → анализирует. Если компиляция занимает секунды (LLVM), цикл замедляется. Cranelift компилирует за миллисекунды — цикл остаётся мгновенным.

## Бенчмарки: 4.4x быстрее Python

Мы измерили производительность на трёх типичных алгоритмических задачах. Сравнение: CPython 3.12 (без оптимизаций) vs Synoema JIT (Cranelift, release build). Время включает JIT-компиляцию:

| Бенчмарк | Python | Synoema JIT | Ускорение |
|----------|--------|-------------|-----------|
| fib(30) — рекурсивный Фибоначчи | 277 мс | 47 мс | **5.9×** |
| gcd (100K итераций) — алгоритм Евклида | 143 мс | 83 мс | **1.7×** |
| collatz (10K чисел) — гипотеза Коллатца | 505 мс | 90 мс | **5.6×** |
| **Среднее** | | | **4.4×** |

Важно: **время Synoema включает JIT-компиляцию**. Чистое время исполнения (без компиляции) ещё быстрее — на уровне C с отключёнными оптимизациями.

Для сравнения: PyPy[^pypy] (JIT для Python) даёт ускорение ~5–10× для вычислительных задач, но требует отдельного runtime и не совместим со многими библиотеками.

[^pypy]: **PyPy** — альтернативная реализация Python с JIT-компиляцией. Ускоряет чистый Python в 5–10 раз, но не совместима со многими C-расширениями (NumPy, TensorFlow). Не подходит как замена CPython для большинства проектов.

## Архитектура pipeline

Вот как выглядит полный pipeline компиляции Synoema:

```
Исходный код (.sno)
  │
  ├─ Lexer (706 строк Rust, 80 тестов)
  │   Токенизация + offside rule[^offside]
  │
  ├─ Parser (1398 строк, 36 тестов)
  │   Pratt parser[^pratt] → AST
  │
  ├─ Type Checker (1453 строки, 42 теста)
  │   Hindley-Milner inference → Typed AST
  │
  ├─ Core IR (969 строк, 26 тестов)
  │   Десахаризация → System F
  │
  └─ Backend (выбор)
      ├─ Interpreter (1314 строк, 46 тестов)
      │   Tree-walking, closures, списки, ADT
      │
      └─ Cranelift JIT (944 строки, 34 теста)
          Нативный x86-64, integers + lists
```

[^offside]: **Offside rule (правило отступов)** — принцип, при котором структура кода определяется отступами, а не скобками. Как в Python: блок кода — это строки с одинаковым отступом. В Synoema отступы определяют `where`-блоки.

[^pratt]: **Pratt parser (парсер Пратта)** — алгоритм синтаксического анализа выражений, придуманный Вотоном Праттом в 1973 году. Элегантно обрабатывает приоритеты операторов (умножение раньше сложения) без рекурсивного спуска. Используется в GCC, V8 (JavaScript), rustc.

7 crates[^crate], 7055 строк Rust, 264 теста, 0 ошибок.

[^crate]: **Crate (крейт)** — единица компиляции и библиотека в языке Rust. Аналог «пакета» или «модуля» в других языках. Проект Synoema состоит из 7 крейтов: lexer, parser, types, core, eval, codegen, repl.

## Что это даёт AI-агентам

Сценарий: AI-агент получает задачу «проанализировать 1M строк лога и найти аномалии».

**С Python:**
```
LLM генерирует Python-скрипт (~200 токенов, 1.5 сек)
→ Запускает через subprocess
→ Python парсит лог (12 секунд)
→ Агент анализирует результат
→ Общее время: ~15 секунд
```

**С Synoema:**
```
LLM генерирует sno-код (~108 токенов, 0.8 сек)
→ JIT-компиляция (50 мс)
→ Нативный код парсит лог (~3 секунды)
→ Агент анализирует результат
→ Общее время: ~4 секунды
```

Экономия: **73% времени**, **46% токенов**, **нулевые зависимости** (не нужен pip install).

## Что дальше

В следующей статье мы разберём **Hindley-Milner** — систему вывода типов, которая даёт 100% type safety без единой аннотации. Это тот механизм, который делает type-guided constrained decoding возможным: компилятор знает типы в каждой точке программы, не требуя от LLM их генерировать.

## Глоссарий

| Термин | Объяснение |
|--------|-----------|
| **JIT-компиляция** | Компиляция «на лету» — код превращается в машинные инструкции прямо перед исполнением |
| **AOT-компиляция** | Компиляция заранее — как в C/Rust. Дольше, но код максимально оптимизирован |
| **Cranelift** | Backend-компилятор на Rust. 10× быстрее LLVM при компиляции, ~86% качества кода |
| **LLVM** | Промышленный стандарт для компиляторов. Используется в Clang, Rust, Swift |
| **Backend** | Часть компилятора, генерирующая машинный код |
| **AST** | Абстрактное синтаксическое дерево — структура данных, представляющая программу |
| **Core IR** | Промежуточное представление — упрощённый внутренний язык компилятора |
| **Agentic workflow** | Подход, где LLM действует как автономный агент: планирует, кодирует, исполняет |
| **Duck typing** | Принцип Python: тип переменной не объявляется, ошибки обнаруживаются только при исполнении |
| **Offside rule** | Структура кода определяется отступами (как в Python), а не скобками |
| **Pratt parser** | Алгоритм разбора выражений с приоритетами операторов |
| **Crate** | Единица компиляции в Rust, аналог библиотеки/пакета |
| **PyPy** | Альтернативный Python с JIT. Быстрее, но не совместим со многими библиотеками |
| **CPython** | Стандартная реализация Python. Интерпретатор, написанный на C |
