# Будущее code generation: от промптов к компиляции

## Что будет, когда LLM станут частью компилятора

Эта серия статей прошла путь от проблемы (токены стоят дорого) через исследование (BPE misalignment, constrained decoding, type inference) к конкретному решению (Synoema). Финальная статья — взгляд вперёд.

## Текущая парадигма: LLM как текстовый генератор

Сегодня LLM генерирует код как **текст**:

```
Промпт → LLM → строка символов → парсер → AST → компилятор → исполнение
```

Модель не «понимает» структуру программы. Она предсказывает следующий токен, ориентируясь на статистические паттерны из обучающего корпуса. Скобки закрываются не потому, что модель знает правила грамматики, а потому, что в обучающих данных после `(` чаще всего встречается соответствующая `)`.

Это работает удивительно хорошо — но создаёт фундаментальные ограничения:

- **Нет гарантий корректности.** Модель может сгенерировать невалидный синтаксис.
- **Избыточность.** Текстовое представление содержит информацию, которую компилятор может вывести сам (типы, скобки, ключевые слова).
- **Хрупкость.** Одна пропущенная скобка или неверный отступ — и всё ломается.

## Следующая парадигма: LLM как часть компилятора

Что если LLM не генерирует текст, а **напрямую создаёт структурное представление программы**?

```
Промпт → LLM + Grammar Constraints + Type Constraints
       → гарантированно валидный AST
       → JIT-компиляция
       → нативное исполнение
```

Это не теория — компоненты уже существуют:

- **Constrained decoding** (XGrammar, Outlines) гарантирует синтаксическую корректность.
- **Type-guided generation** (PLDI 2025) добавляет типовые гарантии.
- **JIT-компиляция** (Cranelift) превращает результат в нативный код за миллисекунды.

Synoema — первый язык, спроектированный под все три компонента одновременно. Но это только начало.

## Пять открытых вопросов

### 1. Может ли LLM выучить Synoema лучше, чем Python?

Интуиция подсказывает «нет» — Python доминирует в обучающих данных. Но есть контраргумент: Synoema **проще**. 7 ключевых слов вместо 35. 33 оператора, каждый — один токен. Детерминированная грамматика без ambiguity.

Гипотеза: fine-tuning[^finetune] на 10K примеров Synoema-кода может дать качество генерации, сравнимое с Python, при 46% меньшем количестве токенов. Это проверяемый эксперимент, и мы планируем его провести.

[^finetune]: **Fine-tuning (дообучение)** — процесс дополнительного обучения уже обученной LLM на специализированных данных. Например, модель, обученная на общем тексте, может быть «дообучена» на коде Synoema, чтобы лучше его генерировать. Требует значительно меньше данных и вычислений, чем обучение с нуля.

### 2. Нужен ли AST-level generation?

Сегодня LLM генерирует *текст*, который потом парсится в AST. Что если модель сразу генерирует узлы AST?

```
Вместо: "fac 0 = 1" (5 текстовых токенов)
Модель генерирует: FuncDef("fac", [PatLit(0)], Lit(1))  (3 структурных токена)
```

Это потенциально эффективнее: меньше токенов, невозможны синтаксические ошибки. Но требует переосмысления архитектуры LLM — текущие модели оптимизированы под последовательную генерацию текста.

### 3. Как масштабировать constraints на контексты 1M+ токенов?

XGrammar работает за near-zero overhead на коротких генерациях. Но при контексте в миллион токенов (что уже поддерживают Gemini и Claude) стоимость PDA-состояний растёт. Нужны новые алгоритмы incremental constraint checking[^incremental].

[^incremental]: **Incremental constraint checking** — проверка ограничений, которая переиспользует результаты предыдущих шагов вместо пересчёта с нуля. Для контекстов в миллионы токенов это критично: пересчёт с нуля на каждом шаге был бы слишком дорогим.

### 4. Может ли один язык быть оптимальным и для LLM, и для человека?

Synoema оптимизирован под BPE. Но читать `? x > 0 -> x : -x` менее привычно, чем `if x > 0: return x else: return -x`. Есть ли формат, который оптимален для обоих?

Возможное решение: **два представления** одной программы. LLM работает с компактным BPE-оптимизированным представлением. Человек видит «развёрнутое» представление с привычным синтаксисом. Трансформация между ними — биекция[^bijection], выполняемая автоматически.

[^bijection]: **Биекция (bijection)** — взаимно однозначное соответствие между двумя множествами. Если представление A биективно к представлению B, то каждая программа в A имеет ровно одно соответствие в B и наоборот. Переход между ними — без потерь.

### 5. Какова роль формальной верификации?

Constrained decoding гарантирует синтаксис. Type inference гарантирует типы. Но ни то, ни другое не гарантирует, что программа **делает то, что нужно**.

Следующий рубеж — интеграция формальной верификации[^verification] в inference pipeline: LLM генерирует код, а constraint-система проверяет не только типы, но и спецификацию (предусловия, постусловия, инварианты).

[^verification]: **Формальная верификация** — математическое доказательство того, что программа соответствует спецификации. Например, для функции сортировки: выход — перестановка входа, и он упорядочен. Сегодня используется в авиации, криптографии и ядре seL4. Для LLM-генерации — область активных исследований.

## Видение: Agentic Computation Pipeline

Собирая всё вместе, вырисовывается архитектура будущего:

```
Человеческий intent (естественный язык)
    ↓
LLM + Grammar Constraints + Type Constraints
    ↓
Гарантированно корректный код (Synoema)
    ↓
JIT-компиляция (Cranelift / LLVM)
    ↓
Нативное исполнение
    ↓
Результат → обратно в LLM для анализа
```

Весь цикл — **< 1 секунды**. Без pip install. Без Docker. Без CI/CD. LLM думает → Synoema формализует → Cranelift исполняет → результат.

Это не замена программистов. Это новый инструмент — как калькулятор не заменил математиков, но изменил то, какие задачи стали решаемыми.

## Заключение серии

За семь статей мы прошли путь:

1. **Проблема** — каждый токен стоит квадратично дорого
2. **BPE** — Python тратит 46% на синтаксический оверхед
3. **Constrained decoding** — можно гарантировать корректность бесплатно
4. **Компиляция** — JIT даёт 4.4× ускорение
5. **Типы** — Hindley-Milner: нулевые аннотации, 100% гарантии
6. **Synoema** — всё вместе, 264 теста, open source
7. **Будущее** — от текстовой генерации к структурной компиляции

Мы верим, что пересечение теории компиляторов, типовых систем и LLM — одна из самых перспективных областей CS. Synoema — первый шаг. Следующие шаги — за вами.
