# Правила проекта Synoema

> Правила вытекают из целей: оптимизация для LLM-генерации, надёжность, производительность.
> Приоритет документов: CLAUDE.md > RULES.md > архитектура/цели > OpenSpec. См. иерархию в CLAUDE.md.

## Цели → Правила

### 1. LLM-оптимизация
- **BPE-alignment:** каждый оператор и ключевое слово — ровно 1 BPE-токен (cl100k_base). Верификация: `tools/bpe-verify/verify_bpe.py`
- **GBNF-грамматика:** синтаксис должен допускать constrained decoding. Новый синтаксис → обновить `tools/constrained/synoema.gbnf`
- **Минимальность синтаксиса:** меньше токенов = дешевле генерация. Каждый новый синтаксический элемент должен обосновать своё существование экономией токенов

### 2. Надёжность
- **Тесты перед кодом:** каждая фича — сначала тест, потом реализация
- **0 warnings, 0 broken tests:** `cargo test` и `cargo build` чистые перед каждым коммитом
- **Пересмотр тестов при доработке:** любое изменение в crate → пересмотреть тесты затронутого модуля. Убедиться, что существующие тесты покрывают новое поведение, добавить недостающие, обновить сломанные
- **Коммит = зелёный CI:** никогда не коммитить с красными тестами
- **Формат коммитов:** `feat: описание` или `fix: описание`

### 3. Производительность
- **Interpreter-first, JIT-second:** новые фичи сначала в interpreter (reference), потом в JIT (performance)
- **Tagged pointer ABI:** JIT-значения используют tagged i64 (bit 0 = list, bit 1 = string, tag-bytes для ADT/float/record). Не ломать ABI
- **Runtime FFI паттерн:** новые JIT-возможности через `extern "C"` в runtime.rs → регистрация в compiler.rs
- **Arena allocator:** все JIT-аллокации через arena, `arena_reset` после каждого запуска

### 4. Язык общения
- **Общение с пользователем:** на русском
- **Размышления (thinking):** на минифицированном английском (краткий, без артиклей, сокращённый)

### 5. Минимализм
- **Зависимости:** только Cranelift + pretty_assertions. Нет tokio, serde, async
- **Код:** идиоматический Rust, без unsafe кроме FFI в runtime.rs
- **Файлы:** не плодить файлы — расширять существующие

### 6. Синтаксические инварианты
- Ternary `?` использует `:` как else-разделитель — тот же символ, что cons. Cons в then-ветке требует скобок: `? cond -> (x : xs) : rest`
- Offside rule: отступы → INDENT/DEDENT токены (как Python, но для функционального языка)
- Pratt parser с 13 уровнями приоритета

### 7. Документация

Каждый новый документ — определи аудиторию и помести в правильную директорию:

| Аудитория | Директория | Что писать |
|-----------|-----------|-----------|
| Человек-пользователь | `docs/user/` | Туториалы, синтаксис с объяснениями, примеры |
| LLM-пользователь | `docs/llm/` | Quick reference, таблицы операторов, аксиомы, GBNF. Стиль: ≤1800 токенов, без prose |
| LLM-разработчик (Claude agent) | `context/` | Rules, Architecture, Project State, Dev Guide, Phases |
| Формальная спецификация | `docs/specs/` | Language reference, grammar, roadmap |
| Научные обоснования | `docs/research/` | Peer-reviewed факты, бенчмарки |
| Outreach/статьи | `docs/articles/` | Образовательная серия для сообщества |
| Internal QA | `docs/` (корень) | testing.md, stress-server.md |

**Правило:** не смешивай аудитории в одном файле. При сомнении — спроси: "кто читатель?"

#### 7a. Актуализация документации

Документация обновляется вместе с изменением, которое её затрагивает. Правило: **коммит с изменением = коммит с обновлённой документацией**.

| Тип изменения | Что обновить обязательно |
|---------------|------------------------|
| Новый синтаксис / оператор | `docs/llm/synoema.md` (таблица операторов), `docs/specs/language_reference.md` (EBNF), `docs/user/syntax.md` если существует |
| Новая фича в interpreter/JIT | `context/PROJECT_STATE.md` (что работает), `context/PHASES.md` (фаза), `CLAUDE.md` (статус если меняются метрики) |
| Новый crate или изменение LOC | `context/ARCHITECTURE.md` (таблица crates + LOC), `context/PROJECT_STATE.md` |
| Новая CLI команда | `CLAUDE.md` (секция Команды), `docs/user/README.md` |
| Изменение ABI / FFI | `context/ARCHITECTURE.md` (Tagged Pointer ABI / Runtime FFI паттерн) |
| Новый MCP инструмент | `docs/mcp.md` |
| Изменение тестов (количество) | `CLAUDE.md` (строка статуса), `docs/testing.md`, `context/PROJECT_STATE.md` |
| Новая зависимость | `context/RULES.md` (секция Минимализм) — только если разрешена |

**LLM-разработчик:** перед завершением задачи пройди по таблице — проверь, затронуто ли что-то из левого столбца.

#### 7b. Требования к LLM-документации (docs/llm/)

Цель: LLM получает максимум информации в минимуме токенов. Каждый файл в `docs/llm/` должен соответствовать:

**Бюджет токенов:**
- Системный промпт: ≤1800 токенов (cl100k_base) на файл
- Inline-вставка: ≤400 токенов (одна конструкция / один раздел)
- Верификация: `python tools/bpe-verify/verify_bpe.py docs/llm/synoema.md`

**Минификация без потери качества — правила:**
1. **Нет prose** — только таблицы, списки, однострочные примеры кода. Запрещены вводные абзацы, пояснения "почему", история
2. **Один пример на конструкцию** — достаточен минимальный рабочий сниппет. Исключение: паттерн-матчинг (2–3 строки для multi-equation)
3. **Негативы явно** — секция "Отсутствует" обязательна: `no if`, `no let`, `no def`, `no return`, `no ,` в списках. LLM должна получить override своих дефолтов
4. **Contrast-таблица** — для каждой не-очевидной конструкции: `Python/Haskell → Synoema` в одну строку
5. **Нет дублирования** — `docs/llm/` не дублирует `docs/specs/`. Ссылка вместо повтора
6. **Проверка полноты** — каждая синтаксическая конструкция языка должна быть представлена хотя бы однострочным примером. Отсутствие = баг документации
7. **Структура фиксирована** — порядок секций: аксиомы → операторы → функции → ADT → typeclasses → модули → IO → отсутствующее → contrast-таблица → gotchas. Не переставлять без веской причины

#### 7c. Структура файлов docs/llm/ как справочника

Цель: LLM не загружает весь справочник в контекст — она запрашивает нужный фрагмент точечно.

**Принцип:** один файл = одна тема = один запрос. Файл должен быть самодостаточным при чтении в изоляции.

**Лимиты на файл:**
- Максимум **400 токенов** (cl100k_base) на файл-фрагмент
- Максимум **600 токенов** для составных тем (операторы, типы)
- `synoema.md` — единственное исключение: сводный индекс ≤1800 токенов, только заголовки + ссылки на фрагменты

**Структура директории:**
```
docs/llm/
  synoema.md          ← индекс: аксиомы + ссылки на все фрагменты (≤1800 токенов)
  operators.md        ← таблица операторов с приоритетами (≤600 токенов)
  functions.md        ← определение функций, pattern matching (≤400 токенов)
  types.md            ← ADT, typeclasses, records (≤600 токенов)
  control.md          ← ternary, local bindings, offside rule (≤400 токенов)
  modules.md          ← mod/use, импорты (≤400 токенов)
  io.md               ← IO, effects, <- sequencing (≤400 токенов)
  stdlib.md           ← встроенные функции, list/string stdlib (≤600 токенов)
  absent.md           ← что отсутствует + contrast-таблица + gotchas (≤400 токенов)
```

**Правила файла-фрагмента:**
1. **Заголовок = тема запроса** — H1 должен совпадать с тем, как LLM будет искать файл (`# Operators`, `# Pattern Matching`)
2. **Нет перекрёстных зависимостей** — файл не предполагает, что LLM прочитала другой файл. Минимальный контекст встроен прямо в файл
3. **Якоря для поиска** — каждый раздел внутри файла снабжён коротким заголовком H2, по которому можно найти конкретную строку через Grep
4. **Нет вступлений** — файл начинается сразу с контента (таблица или пример), не с объяснения "этот файл содержит..."
5. **Индекс synoema.md** — содержит только: 6 аксиом + таблицу "тема → файл → строки". Не дублирует контент фрагментов

**Когда файл нарушает лимит — разбить, не сжимать:** если фрагмент не помещается в 400–600 токенов при сохранении всех конструкций — делить на два файла (например `types-adt.md` + `types-classes.md`), не выбрасывать конструкции.

### 8. Лицензирование и IP

**Многокомпонентная лицензия:** проект использует разные лицензии для разных компонентов.

| Компонент | Лицензия | SPDX |
|-----------|---------|------|
| `lang/crates/` (кроме codegen) | Apache-2.0 | `Apache-2.0` |
| `lang/crates/synoema-codegen/` | BSL-1.1 → Apache-2.0 (36 мес.) | `BUSL-1.1` |
| `tools/` | BSL-1.1 → Apache-2.0 (36 мес.) | `BUSL-1.1` |
| `spec/` | CC-BY-SA-4.0 | `CC-BY-SA-4.0` |
| `docs/` | CC-BY-SA-4.0 | `CC-BY-SA-4.0` |
| `examples/` | MIT-0 | `MIT-0` |

**Правила для SPDX-заголовков:**
- Каждый новый `.rs` файл ОБЯЗАН иметь SPDX-заголовок (первые 2 строки)
- Apache-2.0 файлы: copyright = `Synoema Contributors`
- BSL-1.1 файлы: copyright = `Andrey Bubnov` (licensor)
- Скрипт: `scripts/add_headers.sh` для добавления заголовков

**DCO (Developer Certificate of Origin):**
- Все коммиты должны быть подписаны: `git commit -s`
- Полные правила: `CONTRIBUTING.md`

**Товарные знаки:**
- "Synoema", "sno", расширения `.sno`/`.snm`/`.snp` — TM Andrey Bubnov
- Правила использования: `TRADEMARK.md`
- Форки ОБЯЗАНЫ быть переименованы

**Ключевые документы:**
- `LICENSE` — корневая Apache-2.0 с multi-license заголовком
- `NOTICE` — атрибуции
- `PATENTS` — defensive patent pledge
- `CONTRIBUTING.md` — DCO + правила контрибуции
- `SECURITY.md` — политика раскрытия уязвимостей
- `docs/IP_STRATEGY.md` — стратегия IP (внутренний)
