# Design: Article Series Expansion

## Решения

### 1. Файловая структура
Новые статьи следуют существующему шаблону: `NN_en_slug.md`.
Файлы создаются ТОЛЬКО на английском (EN). RU-версии — отдельный change.

### 2. Обновление числовых данных
Все статьи 4-7 используют актуальные числа из PROJECT_STATE.md:
- 890+ tests → используем "890+" (не округляем до 900)
- ~12,000 LOC → используем "~12,000"
- 8 crates → перечисляем все 8
- Обновляем architecture pipeline с актуальными числами строк и тестов по crate

### 3. Формат бенчмарк-статей (#8-#11)
Каждая новая статья содержит:
- **Methodology** — как измеряли, воспроизводимость
- **Results** — таблицы с данными (или placeholder для нестабильных)
- **Analysis** — что данные значат
- **Limitations** — честное признание ограничений
- **Implications** — практический вывод

### 4. Placeholder-стратегия
- #8 (Token): данные из run_001/run_003 стабильны → использовать
- #9 (Runtime): данные нестабильны → шаблон с `[PLACEHOLDER: stable benchmark results]`
- #10 (LLM Gen): Phase C не запускался → полная методология + template для результатов
- #11 (Cost): формулы + примеры на основе данных Phase A

### 5. Серийная связность
Каждая статья содержит:
- Header: "Part N of *Token Economics of Code*"
- Footer: ссылки на предыдущие и следующую
- Consistent: "Who this is for" blockquote
- Glossary: таблица терминов

### 6. Адаптация под платформы
В рамках данного change создаём ОДНУ версию каждой статьи (Medium-формат как основной).
dev.to адаптация = отдельный change (минимальные различия: убрать footnotes → inline, добавить tags header).

## Зависимости
- Benchmark данные: используем то что есть в `benchmarks/results/`
- Числа проекта: из `context/PROJECT_STATE.md`
- Не зависит от запуска новых бенчмарков
